# change 5: масштабный парсинг + полный pipeline

## что добавлено в parser/listing.py

- `seen_in_run` — set из `cian_id` всех уже-обработанных в этом запуске объявлений. Cian на разных страницах листинга часто отдает одни и те же объявления (особенно ближе к концу выдачи), и без дедупа мы тратим запросы и логируем их повторно. Дедуп по `cian_id` эквивалентен дедупу по url, потому что cian_id извлекается прямо из URL карточки.
- `commit_every` (по умолчанию 50) — коммит в Postgres каждые N апсертов. Раньше коммит был только в конце страницы; при падении посередине терялись все офферы текущей страницы. Теперь после `[commit] saved 50 offers` данные уже в БД, можно безопасно прервать и продолжить.
- Между страницами теперь рандомная пауза `2-5 сек` (`pause_min`/`pause_max`). На больших объемах фиксированная пауза в 3 сек паттернится антиботом.
- Новый сводный лог по странице: `new=24 dup_in_run=4 unique_total=132` — видно, сколько уникальных набралось всего и сколько на этой странице было повторов.

## новая команда в main.py

`pipeline` — один прогон всего процесса с разумными дефолтами:
```
uv run python main.py pipeline --pages 36 --headless
```
36 страниц * 28 объявлений ≈ 1000 шт. Запускает по очереди:
1. `listing` — собирает базовые поля + URL фото
2. `offers` — добирает детали и обновляет фото-список с карточек
3. `photos` — качает по 3 webp на оффер
4. `snapshot.py` — фиксирует результат в DVC

Флаги:
- `--pages N` — сколько страниц листинга
- `--commit-every N` — коммит каждые N апсертов (по умолчанию 50)
- `--headless` — без видимого браузера (быстрее, но cian чаще ловит антибота на голом chromium, поэтому если упадёт — запускай без флага)
- `--no-snapshot` — не делать DVC-снапшот в конце

## зачем нужны DVC-снапшоты — короткое объяснение

БД и фото — бинарные тяжелые артефакты, которые в git тащить нельзя (репо распухнет, история нечитаема). DVC решает это так:
- Сами файлы (`data/cian.sql.gz`, `photos/`) лежат в DVC-кэше и в remote (`../cian-dvc-storage`).
- В git коммитятся только маленькие `.dvc`-метафайлы с MD5-хэшами (~100 байт каждый).
- В любой момент `git checkout <старый_коммит> && uv run dvc checkout` откатывает БД и фото к тому состоянию.
- На другой машине `git clone` + `uv run dvc pull` восстановит данные.
- `uv run dvc diff <commit>` показывает, что добавилось/удалилось.

Это страховка: можно безопасно ронять и пересобирать БД, всегда есть к чему откатиться, плюс можно делить снапшоты с командой через cloud-remote (S3/Yandex и т.п.).

## как запустить большой парсинг

```bash
uv run python main.py pipeline --pages 36 --headless
```

Долго (час+). Если хочется по фазам:
```bash
uv run python main.py listing --pages 36 --headless --commit-every 50
uv run python main.py offers --headless
uv run python main.py photos
uv run python snapshot.py
```

Если упало посередине listing — данные за прошлые страницы уже в БД (благодаря `commit_every=50`), просто перезапусти ту же команду: повторно увиденные `cian_id` отфильтруются дедупом.
